# 00 项目总览：双模块对照设计

> 本仓库 `agentdemo` 包含两个**独立**的 Java 17 Agent 演示应用，刻意保留差异以做教学对照。
> 本文档给出全局视图，细节分发到 `pimono/`、`agentscope/` 子目录，以及数据流、索引、横切关注点等专题文档。

---

## 1. 仓库定位

| 维度 | 说明 |
|---|---|
| 仓库性质 | 教学型 + 生产型双轨演示 |
| 父 POM | `com.demo:agent-demos:1.0.0`，两个子模块互不依赖 |
| JDK | Java 17（vendored 在 `.tools/`，gitignored） |
| 构建 | Maven + `maven-shade-plugin` 产出 uber-jar |
| 框架 | 无 Spring，纯手写 `public static void main` 装配 |
| 入口 | `PiMonoDemoApplication` / `AgentScopeDemoApplication` |
| 协议 | OpenAI 兼容 `/chat/completions` + MCP（JSON-RPC 2.0 over stdio） |

---

## 2. 双模块定位对比

| 对比项 | pimono-demo | agentscope-demo |
|---|---|---|
| 规模 | ~700 行 / 8 主类 / 25 测试 | ~2500 行 / 30+ 主类 / 127 测试 |
| 目标 | 极简原型、教学骨架 | AgentScope 2.0 全套生产级实现 |
| ReAct 循环 | `AgentCore.chat()` 单文件 while 循环 | `Agent.reply()` 跨中间件钩子分布 |
| 上下文管理 | `ContextManager` FIFO 50 条滑窗 | `ContextCompressionMiddleware` 结构化压缩 |
| 中间件 | 无 | 6 钩子洋葱模型（Tracing/Compression/Permission/Budget） |
| 事件流 | 无 | `EventStream` 20+ 类型化事件、可重放 |
| 权限 | 无 | 三引擎三态（Rules→Mode→Built-in / ALLOW·DENY·ASK） |
| 沙箱 | 无 | LocalWorkspace / Docker / E2B 三态 + FilePermissionManager |
| 团队 | 无 | Leader-Worker + 5 个团队工具 |
| 凭证 | 单一 env var | 主备 failover（DefaultCredentialProvider） |
| 内置工具 | 5 个 mock（weather/calculate/search/time/translate） | 4 文件 + 3 代码执行 + 5 团队 |
| UI | JLine3 ANSI REPL | JLine3 REPL + 团队进度跟踪 + 三档 verbosity |

## 3. 通用 Agent 五层范式

两个模块共享同一抽象骨架（在 `ARCHITECTURE.md` 中详述）：

```
┌─────────────────────────────────────────────────────────┐
│  ReAct Loop（推理-行动循环，maxIterations 兜底）          │
├─────────────────────────────────────────────────────────┤
│  LLM Client（OpenAI 兼容 chat/completions）              │
├─────────────────────────────────────────────────────────┤
│  Tool Channel（builtin tools + MCP stdio 外接）          │
├─────────────────────────────────────────────────────────┤
│  Context Store（系统提示词 + 历史消息 + 工具结果）        │
├─────────────────────────────────────────────────────────┤
│  Tool-call Metadata Closure（tool_call_id 必须闭环）     │
└─────────────────────────────────────────────────────────┘
```

| 层 | pimono 实现 | agentscope 实现 |
|---|---|---|
| ReAct | `AgentCore.chat()` | `Agent.reply()` |
| LLM | `ai/LlmClient`（OkHttp+Jackson） | `model/ChatModel`（多后端） |
| MCP | `mcp/McpServerConnection` | `mcp/MCPClient` + 声明式 Stdio/Http 配置 |
| 上下文 | `context/ContextManager` FIFO 滑窗 | `middleware/ContextCompressionMiddleware` 结构化 |
| 元数据 | `Context.ToolCallEntry` | `message/ContentBlock`（6 块类型） |

> **关键不变量**：`tool_call_id` 在两个模块都是强约束，request 与 result 必须一一对应，禁止丢弃或乱序。

## 6. 已知差异/待校准项

- `ARCHITECTURE.md` 写 `MAX_TOOL_ROUNDS=10`，实际 `AgentCore.java` 已改为 `MAX_TOOL_ROUNDS=50`；`Agent.maxIterations=50`。两个模块当前均为 50。
- `ReplyBudgetControlMiddleware` 自 2026-06 起不再抛 `BudgetExceededException`，仅发事件 + 日志；`Agent.maxIterations` 是最终兜底。
- `FilePermissionManager` 与 `PermissionEngine` 是**两套独立**的权限系统：前者管文件路径，后者管工具调用；通过 `SecureFileWorkspace` 桥接。
