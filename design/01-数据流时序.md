# 01 数据流时序：关键流程详解

> 以时序方式拆解两个模块的核心运行流程。pimono 用单文件循环，agentscope 用中间件 + 事件流分层。

---

## 1. pimono：ReAct 主循环

```
用户输入 ──┐
           ▼
   ┌────────────────────────────────────────┐
   │ AgentCore.chat(userInput)              │
   │ 1. context.addUserMsg(userInput)       │
   │ 2. while round < MAX_TOOL_ROUNDS(50):  │
   │    a. llmClient.chatCompletion(messages, tools)  │
   │    b. 若 response 无 tool_calls → 返回  │
   │    c. context.addAssistantMsg(response)│
   │    d. for tc in tool_calls:            │
   │        result = executeTool(tc)        │
   │        context.addToolResult(tc.id, result)│
   │ 3. return 最终文本                      │
   └────────────────────────────────────────┘
```

**关键点**：
- `MAX_TOOL_ROUNDS = 50`（与 `ARCHITECTURE.md` 写的 10 不符，以代码为准）
- `tool_call_id` 通过 `Context.ToolCallEntry` 闭环
- 无中间件、无事件流，所有观察性只能靠日志

## 2. agentscope：完整 reply 流水线

### 2.1 启动装配（main 9 步）

```
main()
  ├─ 1. determinePrimaryProvider（dashscope > openai）
  ├─ 2. ChatModel 构造（OkHttp 客户端）
  ├─ 3. MCPClient 构造 + initialize()
  ├─ 4. SecureFileWorkspace + registerFileTools
  ├─ 5. CodeExecutionManager + registerCodeExecutionTools
  ├─ 6. PermissionEngine（Rules → Mode → Built-in）
  ├─ 7. WorkspaceManager
  ├─ 8. Leader Agent 构造 + 中间件链挂载
  └─ 9. AgentTeam + registerTeamTools → REPL
```

### 2.2 单次 reply 时序

```
REPL → AgentTeam.reply(userInput)
        │
        ▼
   Agent.reply(userInput)
   │
   ├─ progressTracker.onReplyStart
   ├─ ctx = AgentContext(...)
   ├─ middlewareChain.fireReplyStart(ctx)        ← onReplyStart 钩子
   ├─ context.add(userMsg)
   ├─ eventStream.emit(ReplyStart)
   │
   │  ┌─── for iteration in [0, maxIterations=50): ──────────┐
   │  │                                                       │
   │  │  progressTracker.onModelCallStart                     │
   │  │  messages = buildMessages()                           │
   │  │  tools = mcpClient.listTools()                        │
   │  │  middlewareChain.fireModelCall(ctx, req)              │
   │  │  response = chatModel.chat(provider, messages, tools) │
   │  │  progressTracker.onModelCallComplete(usage)           │
   │  │  middlewareChain.fireModelCallEnd(ctx, response)      │
   │  │                                                       │
   │  │  context.add(response)                                │
   │  │                                                       │
   │  │  ── if !response.hasToolCalls(): break（最终答案）   │
   │  │                                                       │
   │  │  ┌── for toolCall in response.getToolCalls(): ───┐   │
   │  │  │  progressTracker.onToolCall(name)              │   │
   │  │  │  middlewareChain.fireToolCall(ctx, tc)         │   │
   │  │  │                                                  │   │
   │  │  │  decision = permissionEngine.check(name, args) │   │
   │  │  │  ── if DENY: 注入"权限拒绝"结果，continue      │   │
   │  │  │                                                  │   │
   │  │  │  result = mcpClient.executeTool(name, args)    │   │
   │  │  │  result = truncateToolResult(content, name)    │   │
   │  │  │  context.add(toolResultMsg)                    │   │
   │  │  │  middlewareChain.fireToolResult(ctx, block)    │   │
   │  │  │  eventStream.emit(ToolResult)                  │   │
   │  │  └─────────────────────────────────────────────┘   │
   │  └───────────────────────────────────────────────────┘
   │
   ├─ ctx.setAttribute("contextMessages", context)
   ├─ middlewareChain.fireReplyEnd(ctx, eventStream)        ← onReplyEnd（含压缩）
   ├─ eventStream.emit(ReplyEnd)
   ├─ progressTracker.onReplyComplete(duration)
   └─ return finalResponse
```

### 2.3 中间件洋葱（按声明顺序）

```
TracingMiddleware                → 日志
ContextCompressionMiddleware     → onReplyEnd 时压缩
PermissionMiddleware             → 工具调用前检查（与 PermissionEngine 并行）
ReplyBudgetControlMiddleware     → onModelCallEnd 累计 token，超限发事件
```

注意：`PermissionMiddleware` 与 `Agent.reply()` 内部的 `permissionEngine.check()` 是**两道**检查；后者是硬阻断（DENY 直接注入错误结果）。

## 3. agentscope：Leader-Worker 团队流

### 3.1 创建工作者（agent_create）

```
Leader 调用 agent_create 工具（参数 name, system_prompt）
        │
        ▼
   AgentTeam.createWorker(name, prompt)
   ├─ new MCPClient + initialize()       ← 工作者独立 MCP
   ├─ mcpClient.copyToolsTo(workerMcpClient, 排除 5 个团队工具)
   ├─ new Agent(name, prompt, ..., workerMcpClient, ...)
   ├─ workers.put(name, worker)
   ├─ progressTracker.onLeaderCreateWorker(name)
   └─ 返回 "工作者已创建，ID: xxx"
```

**关键隔离**：工作者拿不到 `agent_create` / `agent_message` / `agent_list` / `team_dissolve` / `agent_message_parallel`，避免越权操作团队。

### 3.2 分发任务（agent_message）

```
Leader 调用 agent_message（worker_name, message）
        │
        ▼
   AgentTeam.sendMessageToWorker(name, message)
   ├─ progressTracker.onLeaderAssignTask / onAgentCommunication / onWorkerStart
   ├─ worker.reply(message)    ← 递归进入 §2.2 子流程
   ├─ progressTracker.onWorkerComplete(success, duration)
   └─ 返回 worker 的回复文本
```

工作者的 `reply()` 是完整的 ReAct 循环，可继续调用文件/代码工具。

### 3.3 并行分发（agent_message_parallel）

```
Leader 调用 agent_message_parallel({workerA: msgA, workerB: msgB, ...})
        │
        ▼
   sendMessageToWorkersParallel(tasks)
   ├─ for each task: CompletableFuture.supplyAsync(() -> sendMessageToWorker(...))
   ├─ CompletableFuture.allOf(...).join()       ← 等待全部完成
   └─ 收集 results: Map<name, Msg>
```

### 3.4 解散（team_dissolve）

```
Leader 调用 team_dissolve
   ├─ for worker in workers: worker.shutdown()
   ├─ workers.clear()
   ├─ active = false
   └─ 返回 "团队已解散，关闭 N 个工作者"
```

## 4. agentscope：上下文压缩流

```
onReplyEnd 触发
        │
        ▼
   ContextCompressionMiddleware.onReplyEnd(ctx, stream)
   ├─ context = ctx.getAttribute("contextMessages")
   ├─ if context.size() > THRESHOLD(40):
   │    ├─ recent = context.tail(KEEP_RECENT=10)
   │    ├─ old = context.head(size - KEEP_RECENT)
   │    ├─ summary = summarize(old)   ← 结构化: task_goal/current_state/key_findings/next_steps
   │    └─ context = [summary] + recent
   └─ 否则跳过
```

阈值与保留数：
- `DEFAULT_THRESHOLD = 40` 条消息
- `DEFAULT_KEEP_RECENT = 10` 条

## 5. agentscope：权限检查流（双闸）

### 5.1 工具级权限（PermissionEngine）

```
工具调用发生
   ├─ PermissionMiddleware.onToolCall（中间件层，可选放行）
   │
   ▼
Agent.reply 内部 check(name, args)
   ├─ checkRules：先跑用户规则（bypassImmune 规则在 BYPASS 模式仍生效）
   ├─ checkMode：EXPLORE 只读 / DONT_ASK（ASK→DENY）/ BYPASS（全放行）
   └─ checkBuiltIn：WRITE_TOOL_PREFIXES、DANGEROUS_COMMAND_PATTERNS、DANGEROUS_PATH_PATTERNS
        │
        ▼
   任一非 ALLOW 即终止 → 返回 DENY/ASK
```

### 5.2 文件级权限（FilePermissionManager）

```
SecureFileWorkspace.readFile/writeFile/editFile/listFiles
        │
        ▼
   FilePermissionManager.check/checkList
   ├─ 1. 路径安全（防 ../ 穿越）
   ├─ 2. 黑名单路径（.env / secrets / .git）
   ├─ 3. 黑名单扩展（exe / sh / bat）
   ├─ 4. 白名单扩展
   ├─ 5. 白名单路径
   ├─ 6. 文件大小 ≤ 10MB（list 跳过）
   └─ 7. 默认策略（DENY_ALL 兜底）
        │
        ▼
   任一失败 → FilePermissionDeniedException
```

## 6. agentscope：事件流类型

完整事件序列（一次成功 reply）：

```
ReplyStart
  └─ ModelCall
       ├─ ThinkingBlock     (若 LLM 返回思考块)
       ├─ TextBlock         (流式/最终文本)
       ├─ ToolCall
       │    └─ ToolResult
       └─ ... (循环)
  └─ ReplyEnd
```

异常分支：`Error` 事件、`ReplyBudgetExceeded` 事件（token 超预算但**不中断**）。

---

## 7. pimono vs agentscope：循环复杂度

| 维度 | pimono | agentscope |
|---|---|---|
| 同步循环层数 | 1（while + for tools） | 1（for iteration + for tools） |
| 钩子/事件点 | 0 | 6 中间件钩子 + 20+ 事件类型 |
| 权限检查点 | 0 | 双闸（中间件 + 引擎） |
| 上下文压缩 | FIFO 50 条硬截断 | 结构化压缩（40 阈值） |
| Token 预算 | 无 | 500K 默认，超限仅告警 |
| 工具结果截断 | 无 | 按工具分级（20K/5K/500 字符） |
| 可观察性 | 仅日志 | 日志 + 事件流 + 进度跟踪 |
