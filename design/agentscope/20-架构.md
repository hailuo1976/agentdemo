# agentscope-demo 架构：AgentScope 2.0 六件套生产级实现

> 目标：用 Java 17 全量重写 AgentScope 2.0（Python）的"六件套"架构，演示生产级 Agent 系统的完整形态。
> 规模：~2500 行 / 30+ 主类 / 127 测试。

---

## 2. 装配顺序（main 9 步）

`AgentScopeDemoApplication.main()` 是唯一装配点，必须**自底向上**：

```
1. determinePrimaryProvider(dashscope > openai)
2. new ChatModel(...)             ← LLM 客户端
3. new MCPClient() + initialize()
4. SecureFileWorkspace + MCPClient.registerFileTools(ws)
5. CodeExecutionManager + MCPClient.registerCodeExecutionTools(em)
6. PermissionEngine (Rules → Mode → Built-in, 16 默认规则)
7. WorkspaceManager
8. new Agent(leader, ...)
   └─ middlewareChain 挂载：Tracing / Compression / Permission / Budget
9. AgentTeam + registerTeamTools
   └─ runREPL
```

**修改装配时必须保持此顺序**——上层依赖下层的引用。

## 4. 与 pimono 的本质差异

| 维度 | pimono | agentscope |
|---|---|---|
| 抽象层次 | 1 层（AgentCore 包办） | 6+ 层（六件套） |
| 中间件 | 无 | 洋葱模型 + 6 钩子 |
| 事件 | 无 | 类型化、可重放 |
| 权限 | 信任环境 | 双闸（工具 + 文件） + 三态（ALLOW/DENY/ASK） |
| 沙箱 | 无 | Local / Docker / E2B + FilePermission |
| 凭证 | 单 env | 主备 failover |
| 团队 | 无 | Leader-Worker + 5 工具 + 并行分发 |
| 上下文 | FIFO 50 条 | 结构化压缩（40 阈值，保留 10） |
| Token 预算 | 无 | 500K 默认，超限仅告警 |
| 工具结果 | 全量注入 | 按工具分级截断 |
| 异常处理 | 上抛到 REPL | try/catch 兜底返回错误消息 |

## 5. 中间件洋葱（6 钩子）

```
onReplyStart(ctx)
   ↓
onModelCall(ctx, requestMsg)
   ↓
[chatModel.chat]
   ↓
onModelCallEnd(ctx, response)   ← Budget 在此累计 token
   ↓
onToolCall(ctx, toolCall)       ← Permission 在此检查（软）
   ↓
[mcpClient.executeTool]
   ↓
onToolResult(ctx, resultBlock)
   ↓
onReplyEnd(ctx, eventStream)    ← Compression 在此触发
```

内置 4 个中间件（按声明顺序）：
1. `TracingMiddleware`（日志）
2. `ContextCompressionMiddleware`（阈值 40 / 保留 10）
3. `PermissionMiddleware`（工具调用前置检查）
4. `ReplyBudgetControlMiddleware`（500K 默认，超限仅告警）

## 6. 双权限系统（关键！）

仓库有**两套独立**的权限系统，通过 `SecureFileWorkspace` 桥接：

### 6.1 工具级权限：`PermissionEngine`
- 对象：工具调用（execute_python / read_file / agent_create ...）
- 流程：Rules → Mode → Built-in（任一非 ALLOW 终止）
- 三态：ALLOW / DENY / ASK
- 三模式：EXPLORE（只读）/ DONT_ASK（ASK→DENY，默认）/ BYPASS（全放行）

### 6.2 文件级权限：`FilePermissionManager`
- 对象：文件路径（read_file/write_file/edit_file/list_files）
- 流程：7 步检查（路径安全 → 黑名单路径 → 黑名单扩展 → 白名单扩展 → 白名单路径 → 大小 → 默认策略）
- 默认：`DENY_ALL` 兜底

### 6.3 桥接
`SecureFileWorkspace` 同时持有 `LocalWorkspace`（真实 FS）和 `FilePermissionManager`；
MCP 文件工具通过 `SecureFileWorkspace` 访问 FS，每次调用都经过权限检查。

## 7. 团队协作模型

Leader Agent 拥有 5 个团队工具：

| 工具 | 作用 |
|---|---|
| `agent_create` | 创建工作者（独立 MCPClient） |
| `agent_message` | 向单个工作者发消息（同步等回复） |
| `agent_message_parallel` | 并行向多个工作者发消息 |
| `agent_list` | 列出工作者 |
| `team_dissolve` | 解散团队 |

**隔离**：工作者通过 `MCPClient.copyToolsTo(workerMcpClient, excludeTeamTools)` 拿到的是**剔除团队工具**的工具集，不能越权操作团队。

---

## 8. 工具集分类

| 类别 | 数量 | 工具 |
|---|---|---|
| 文件 | 4 | read_file / write_file / edit_file / list_files |
| 代码执行 | 3 | execute_python / execute_command / install_package |
| 团队（仅 Leader） | 5 | agent_create / agent_message / agent_message_parallel / agent_list / team_dissolve |
| 外部 MCP | N | 通过 `MCP_SERVERS` 配置 |
